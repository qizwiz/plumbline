ORACLE / PRICE FEED — - [ORACLE] TSwapPool::getPriceOfOneWethInPoolTokens — uses spot balanceOf for price — flash-loan or sandwich attack can manipulate price oracle - [ORACLE] TSwapPool::getPriceOfOnePoolTokenInWeth — uses spot balanceOf for price — same manipulation vector as above - [ORACLE] TSwapPool::getOutputAmountBasedOnInput — no staleness check — relies on current reserves which can be stale after paused state - [ORACLE] TSwapPool::swapExactInput — reads balanceOf before transfer — inputReserves is stale, doesn't include the incoming inputAmount

ARITHMETIC — - [ARITHMETIC] TSwapPool::getOutputAmountBasedOnInput — division truncation — `numerator / denominator` floors, user receives less output - [ARITHMETIC] TSwapPool::getInputAmountBasedOnOutput — division truncation — `((inputReserves * outputAmount) * 10000) / ((outputReserves - outputAmount) * 997)` rounds down, protocol receives less input - [ARITHMETIC] TSwapPool::deposit — liquidityTokensToMint division — `(wethToDeposit * totalLiquidityTokenSupply()) / wethReserves` truncates, user minted fewer LP tokens than fair share - [ARITHMETIC] TSwapPool::withdraw — wethToWithdraw/poolTokensToWithdraw division — both truncate, user receives less than proportional share - [ARITHMETIC] TSwapPool::getPoolTokensToDepositBasedOnWeth — division truncation — `(wethToDeposit * poolTokenReserves) / wethReserves` rounds down, deposit ratio mismatch - [ARITHMETIC] TSwapPool::getInputAmountBasedOnOutput — potential overflow on `(inputReserves * outputAmount) * 10000` — large reserves * outputAmount could overflow uint256 - [ARITHMETIC] TSwapPool::_swap — outputToken.safeTransfer 1e18 extra tokens — hardcoded 1e18 ignores token decimals, breaks for non-18-decimal tokens

TOKEN ACCOUNTING — - [TOKEN ACCOUNTING] TSwapPool::swapExactInput — reserves read before transfer — `inputReserves = inputToken.balanceOf(address(this))` doesn't include inputAmount user is about to send, breaks invariant - [TOKEN ACCOUNTING] TSwapPool::swapExactOutput — same issue — reserves read before transfer, calculation uses wrong base - [TOKEN ACCOUNTING] TSwapPool::_swap — extra token incentive sent before input received — CEI violation, outputToken balance changes before inputToken received, breaks invariant - [TOKEN ACCOUNTING] TSwapPool::_swap — 1e18 extra token assumes 18 decimals — if outputToken has 6 decimals (USDC), sends 1e18 base units = massive overpayment - [TOKEN ACCOUNTING] TSwapPool::deposit — no fee-on-transfer check — if poolToken or wethToken has transfer fee, balanceOf won't match expected, ratio broken - [TOKEN ACCOUNTING] TSwapPool::withdraw — burns before transfer — if transfer fails (paused token), liquidity tokens already burned, user loses funds - [TOKEN ACCOUNTING] TSwapPool::sellPoolTokens — wrong parameter order — passes poolTokenAmount as outputAmount to swapExactOutput, but should be inputAmount - [TOKEN ACCOUNTING] PoolFactory::createPool — no check if tokenAddress is valid ERC20 — calling name() on non-ERC20 will revert, but after pool created

EXTERNAL CALLS & REENTRANCY — - [REENTRANCY] TSwapPool::_swap — outputToken transfer before inputToken transfer — if outputToken is malicious ERC777 with hook, can reenter and drain pool - [REENTRANCY] TSwapPool::_swap — swap_count increment before output transfer — malicious token hook can reenter, get multiple 1e18 bonuses in one tx - [REENTRANCY] TSwapPool::deposit — _mint before safeTransferFrom — LP tokens minted before tokens received, user can reenter with LP tokens before providing liquidity - [REENTRANCY] TSwapPool::withdraw — safeTransfer after _burn — correct CEI, but if token has hook, user can reenter other functions with withdrawn funds - [REENTRANCY] TSwapPool::_addLiquidityMintAndTransfer — event emitted with wrong parameter order — `emit LiquidityAdded(msg.sender, poolTokensToDeposit, wethToDeposit)` has poolTokens and weth swapped
