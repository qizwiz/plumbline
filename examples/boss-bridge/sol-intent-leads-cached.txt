ACCESS CONTROL & INIT — - [ACCESS CONTROL] L1Vault::constructor — vault owner is msg.sender not bridge — vault is owned by deployer initially, then transferred; if transfer fails or is forgotten, bridge has no control - [ACCESS CONTROL] L1BossBridge::setSigner — no validation that account != address(0) — could accidentally set zero address as valid signer - [ACCESS CONTROL] L1Vault::approveTo — no validation target != address(0) — could approve zero address, wasting approval slot - [ACCESS CONTROL] TokenFactory::deployToken — no check for deployment success — if create returns address(0), it's stored as valid token address - [ACCESS CONTROL] L1BossBridge::sendToL1 — anyone can call with valid signature — no msg.sender validation; attacker can replay valid signature before intended recipient - [ACCESS CONTROL] L1BossBridge::withdrawTokensToL1 — no nonce/replay protection — same signature can be used multiple times to drain vault
ARITHMETIC — - [ARITHMETIC] L1BossBridge::depositTokensToL2 — vault balance + amount could overflow — if vault.balanceOf near max uint256, addition could wrap - [ARITHMETIC] L1BossBridge::DEPOSIT_LIMIT — hardcoded to 100k ether but no decimals validation — if token has != 18 decimals, limit is wrong scale
TOKEN ACCOUNTING — - [TOKEN ACCOUNTING] L1BossBridge::withdrawTokensToL1 — uses transferFrom with vault as source — assumes vault has given bridge approval; if approval revoked/reduced, withdrawal fails - [TOKEN ACCOUNTING] L1BossBridge::depositTokensToL2 — no check that amount > 0 — allows zero-amount deposits that emit misleading events - [TOKEN ACCOUNTING] L1BossBridge::sendToL1 — arbitrary call can target vault.token — attacker can craft message to call token.approve or transferFrom from vault - [TOKEN ACCOUNTING] L1Vault::approveTo — infinite approval to bridge — if bridge compromised, entire vault drained - [TOKEN ACCOUNTING] L1BossBridge::depositTokensToL2 — checks vault balance after transfer — front-running or direct vault transfers can bypass DEPOSIT_LIMIT
ARRAY / INDEX / INPUT VALIDATION — - [INPUT VALIDATION] L1BossBridge::depositTokensToL2 — no validation from != address(0) — could emit event with zero address - [INPUT VALIDATION] L1BossBridge::depositTokensToL2 — no validation l2Recipient != address(0) — L2 mint could go to burn address - [INPUT VALIDATION] L1BossBridge::withdrawTokensToL1 — no validation to != address(0) — tokens could be sent to burn address - [INPUT VALIDATION] L1BossBridge::withdrawTokensToL1 — no validation amount > 0 — wasteful signature verification for zero withdrawal - [INPUT VALIDATION] TokenFactory::deployToken — no validation symbol not empty — can deploy tokens with empty symbol - [INPUT VALIDATION] L1BossBridge::sendToL1 — no validation target != address(this) — could call bridge's own functions recursively
EXTERNAL CALLS & REENTRANCY — - [REENTRANCY] L1BossBridge::sendToL1 — arbitrary external call before state changes — attacker can reenter if target is malicious contract - [REENTRANCY] L1BossBridge::depositTokensToL2 — emits event after external transfer — reentrancy during safeTransferFrom could cause double-event emission - [EXTERNAL CALLS] L1BossBridge::sendToL1 — arbitrary call with arbitrary data and value — signer can execute any function on any contract as bridge - [EXTERNAL CALLS] L1BossBridge::sendToL1 — uses low-level call without gas limit — could consume all gas or be exploited via gas griefing - [EXTERNAL CALLS] L1BossBridge::withdrawTokensToL1 — calls transferFrom via sendToL1 — if token has callback (ERC777), reentrancy possible despite nonReentrant on sendToL1
CROSS-CHAIN / BRIDGE — - [CROSS-CHAIN] L1BossBridge::depositTokensToL2 — no chain-id in Deposit event — L2 service could pick up event from wrong chain/fork - [CROSS-CHAIN] L1BossBridge::withdrawTokensToL1 — signature doesn't include chain-id — valid signature on one chain valid on all chains/forks - [CROSS-CHAIN] L1BossBridge::sendToL1 — no nonce in message — replay attack possible; same withdrawal signature reusable infinitely - [CROSS-CHAIN] L1BossBridge — deploying on ZKSync with TokenFactory — ZKSync uses different CREATE/CREATE2 behavior; deployed addresses will differ - [CROSS-CHAIN] TokenFactory::deployToken — uses CREATE not CREATE2 on ZKSync — address on L1 != address on L2 for same symbol/bytecode - [CROSS-CHAIN] L1BossBridge::withdrawTokensToL1 — no destination chain validation in signature — signer could intend Ethereum but signature used on ZKSync
ECONOMIC / INCENTIVE LOGIC — - [ECONOMIC] L1BossBridge::DEPOSIT_LIMIT — public mutable variable set in storage — owner could change it (no setter exists but variable not constant/immutable) - [ECONOMIC] L1BossBridge::depositTokensToL2 — DEPOSIT_LIMIT is cumulative vault balance — users can't deposit once limit hit even if others withdrew - [ECONOMIC] L1BossBridge::sendToL1 — no withdrawal limit like deposit — asymmetric; attacker can drain vault if they get signer key - [ECONOMIC] L1BossBridge — signature-based withdrawal is permissionless — anyone observing valid signature can front-run and redirect withdrawal to themselves via sendToL1 - [ECONOMIC] L1BossBridge::withdrawTokensToL1 — encodes transferFrom with vault as sender — bridge needs approval from vault; if signer signs message calling vault.approveTo, bridge loses access - [ECONOMIC] L1BossBridge — no fee mechanism — no incentive for bridge operator; unsustainable model - [ECONOMIC] L1BossBridge::sendToL1 — can send ETH via value parameter — bridge has no receive/fallback; where does ETH come from? - [ECONOMIC] L1Vault — holds tokens but no emergency withdraw — if bridge paused indefinitely, funds locked forever
[SIGNATURE REPLAY] L1BossBridge::sendToL1 — signature has no nonce, expiry, or used-tracking — infinite replay attack; attacker can drain vault with single valid signature
[SIGNATURE MALLEABILITY] L1BossBridge::sendToL1 — uses ECDSA.recover without checking s value — signature malleability possible (though OpenZeppelin ECDSA mitigates this)
[ARBITRARY EXECUTION] L1BossBridge::sendToL1 — signer can execute arbitrary calls as bridge — signer can call vault.approveTo to approve attacker, then steal all tokens
[CENTRALIZATION] L1BossBridge — single owner controls pause and signers — owner can pause indefinitely, trapping funds; or add malicious signer
[VAULT OWNERSHIP] L1Vault — created with msg.sender as owner then should transfer to bridge — if ownership transfer not done atomically in constructor, bridge may not own vault
[CONSTRUCTOR ISSUE] L1BossBridge::constructor — vault created but ownership not transferred — vault.owner() is bridge deployer, not bridge contract itself
[WRONG FROM] L1BossBridge::depositTokensToL2 — takes 'from' parameter but caller could be different — allows anyone to deposit from any address with approval
[NO REFUND] L1BossBridge::depositTokensToL2 — if deposit fails on L2, no refund mechanism — user loses L1 tokens with no L2 credit
[ETH HANDLING] L1BossBridge::sendToL1 — sends ETH via value but bridge has no ETH source — call will fail if value > 0 and bridge has no ETH balance
[DECIMALS MISMATCH] L1Token — 18 decimals assumed everywhere — if TokenFactory deploys token with different decimals, DEPOSIT_LIMIT and accounting breaks
