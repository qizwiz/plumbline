# ANSWER KEY — Cyfrin Puppy Raffle audit findings

Canonical source: Cyfrin/4-puppy-raffle-audit `audit-data` branch,
`audit-data/2023-09-01-puppy-raffle.md`. Curated 2026-06-05 for plumbline reps.

Do NOT feed this file to sol_intent — it is ground truth for `sol_match` only.

## H-1 Reentrancy attack in PuppyRaffle::refund allows entrant to drain contract balance
`refund` sends ETH before zeroing `players[playerIndex]`. An attacker contract whose `fallback`/`receive` re-enters `refund` can drain the contract balance one ticket value per re-entry. Faithful fix: zero the slot before transferring (CEI), or apply a reentrancy guard.

## H-2 Weak randomness in PuppyRaffle::selectWinner allows anyone to choose winner
`selectWinner` derives randomness from `keccak256(abi.encodePacked(msg.sender, block.timestamp, block.difficulty))`. All inputs are miner/sender-influenced, so the outcome is predictable / griefable. Faithful fix: Chainlink VRF or commit-reveal.

## H-3 Integer overflow of PuppyRaffle::totalFees loses fees
`totalFees` is `uint64`; cumulative fees can exceed `2**64 - 1` (~18.4 ETH) and silently wrap to zero. After overflow, `withdrawFees` either reverts on the strict balance check (H-4 / M-2) or pays a wrong amount. Faithful fix: use `uint256`.

## H-4 Malicious winner can forever halt the raffle
`selectWinner` `_safeMint`s an NFT to a winner who might be a contract without `onERC721Received`. The mint reverts, the prize never transfers, fee/winner state never advances, the raffle is bricked. Faithful fix: pull-pattern claim, or `_mint` (not `_safeMint`) for NFT payout.

## M-1 enterRaffle O(n²) duplicate-address loop is a potential DoS vector
The nested duplicate check in `enterRaffle` is O(n²) over current players. Cost grows quadratically; an attacker who pre-fills the list raises the gas cost for everyone else. Faithful fix: a `mapping(address => bool) hasEntered` instead of array scan.

## M-2 Balance check on withdrawFees enables griefer to block withdrawals via selfdestruct
`withdrawFees` requires `address(this).balance == uint256(totalFees)`. A griefer can `selfdestruct` ETH into the raffle, breaking equality, blocking withdrawal forever. Faithful fix: drop the strict equality check; just transfer `totalFees`.

## M-3 Unsafe cast of fee loses fees
`totalFees = totalFees + uint64(fee)` casts a `uint256` fee down to `uint64`. Per-raffle fees exceeding `2**64 - 1` wei lose the high bits silently. Faithful fix: widen `totalFees` (H-3 fix) and remove the cast.

## M-4 Smart-contract winners without receive/fallback halt the raffle prize transfer
The ETH payout in `selectWinner` is `winner.sendValue(prizePool)`. A contract winner without `receive`/`fallback` reverts the call, blocking raffle reset (related to H-4). Faithful fix: pull-pattern claim, or skip-and-reassign.

## I-1 Floating pragma
`pragma solidity ^0.7.6;` should be pinned, not floating.

## I-2 Magic numbers
Literals like `80`, `20`, `100` for fee/prize math should be named constants.

## I-3 Test coverage
Coverage gaps in `selectWinner`, `withdrawFees`, `getActivePlayerIndex`.

## I-4 Zero-address validation
`feeAddress` is set without a `address(0)` check; can brick fees.

## I-5 `_isActivePlayer` is unused dead code and should be removed
Unused internal helper.

## I-6 Unchanged variables should be constant or immutable
`raffleDuration` etc. never change post-deploy; mark `immutable`.

## I-7 Potentially erroneous active player index
`getActivePlayerIndex` returns `0` for a non-existent player, which collides with the legitimate index `0`. A refund attempt at the returned `0` may zero out the wrong slot.

## I-8 Zero address may be erroneously considered an active player
Related to I-7 — `address(0)` reads as the "no entry" sentinel but can shadow a real entry at index 0.
