- LibBytes.sol:138 — readRSVCompact function — `uint256 yParity = uint256(yParityAndS >> 255)` extracts top bit, then `s = bytes32(uint256(yParityAndS) & ((1 << 255) - 1))` masks lower 255 bits — if `yParityAndS` has malformed high bits, `yParity` could be non-zero/non-one, leading to `v = uint8(yParity) + 27` being invalid (28+), causing ecrecover to fail silently.

- SessionSig.sol:203 — _decodePermissions — reads `uint8` length but creates array without checking if remaining bytes can hold that many permissions — if encoded data is truncated, reading beyond bounds returns dirty memory values as fake permissions.

- Attestation.sol:48 — fromPacked — reads `uint24 dataSize` for applicationData, then does `encoded[newPointer:newPointer + dataSize]` — if dataSize exceeds remaining encoded.length, slice operation doesn't revert but returns shorter slice, causing silent truncation of applicationData.

- Permission.sol:67 — readPermission — reads `uint8 rulesLength` and creates array, then loops to populate — if encoded data ends mid-rule, subsequent `readUint8`/`readBytes32` calls read beyond calldata returning zeros, creating phantom rules with zero values.

- WebAuthn.sol:75 — verify function — `add(0x14, t), n` checks `t + 0x14 < n` but uses `lt(add(0x14, t), n)` which would pass if `t + 0x14` overflows uint256 wrapping to small value — attacker could craft `t` near uint256.max to bypass length check.

- WebAuthn.sol:90 — verify function — `add(q, c)` computes challenge end position but if `q + c` overflows (q is length + 13, c is challengeIndex), overflow wraps and check `lt(add(q, c), n)` passes incorrectly — allows reading challenge past clientDataJSON end.

## ACCESS CONTROL & INIT

- [REVISED] GROUND: ExpectedSigner = address(this) (the current wallet); the signed payload is represented by `_imageHash` (the configuration hash); WalletIdentity is missing from the signature validation — `_imageHash` alone does not bind the signature to `address(this)`, only to a configuration that may be shared by multiple wallets deployed via the same FACTORY with the same salt.

VIOLATION: invariant SignatureWalletBinding fails — the validation accepts `_imageHash` from any wallet where `keccak256(abi.encodePacked(hex"ff", FACTORY, _imageHash, INIT_CODE_HASH)) == address(some_wallet)`, not necessarily `address(this)`. If two wallets share the same `_imageHash` (same configuration/salt), a signature valid for wallet A is accepted by wallet B.

ATTACK: Deploy two wallets via FACTORY with identical `_imageHash`; obtain a valid signature from wallet A; replay that signature against wallet B's `_isValidImage` — it passes because the imageHash→address derivation is deterministic but does not enforce that the derived address equals `address(this)`.

- [ACCESS CONTROL] Implementation::_setImplementation Implementation.sol:37 — stores implementation address at storage slot `address()` (slot 0) — this collides with any other contract state using slot 0, and specifically collides with the proxy pattern where slot 0 might store the implementation; a malicious implementation could overwrite this during delegatecall execution.

## CROSS-CHAIN / BRIDGE

- [CROSS-CHAIN] Payload::domainSeparator Payload.sol:37-47 — uses `block.chainid` in domain separator unless `_noChainId` flag is set — but `_noChainId` flag is set *during* signature validation (BaseAuth.sol:76), not at signature creation time; cross-chain replay possible if attacker captures a signature with `noChainId=false` on chain A and replays on chain B where validation sets `noChainId=true`.

- [CROSS-CHAIN] Payload::hashFor Payload.sol:124 — computes hash for a specific `_wallet` address, used in `FLAG_SIGNATURE_ANY_ADDRESS_SUBDIGEST` — but does not include chain ID when `_noChainId=true`; cross-chain deployed wallets at same address (via CREATE2) accept same signatures across chains, enabling replay attacks.

- [REVISED] **GROUND:** The TLA+ spec's `WalletIdentity` field corresponds to the `address` argument in `_payload.hashFor(address)` at BaseSig.sol:367-374. The code path accepts `_payload.hashFor(address(0))` when `FLAG_SIGNATURE_ANY_ADDRESS_SUBDIGEST` is set, omitting the actual wallet's `address(this)` from the hash. `SignatureWeight` = `type(uint256).max` is assigned upon match.

**VIOLATION:** Invariant `SignedPayloadBindsWallet` is violated — the signature payload hashes with `address(0)` instead of the verifying wallet's own address, allowing a single signature to be valid across multiple wallet instances at different addresses (and chains).

**ATTACK:** Attacker generates signature over `_payload.hashFor(address(0))` with `FLAG_SIGNATURE_ANY_ADDRESS_SUBDIGEST` set; submits this signature to any wallet instance on any chain where that wallet is deployed at the same address; all instances accept it with max weight, enabling cross-wallet replay.

- [ECONOMIC] BaseAuth::signatureValidation BaseAuth.sol:76-80 — static signature with expired `timestamp <= block.timestamp` reverts — but if attacker front-runs with block timestamp manipulation (e.g., via miner), legitimate static signatures can be invalidated, forcing fallback to full signature validation and higher gas costs or DoS.

- [ECONOMIC] BaseSig::recoverBranch BaseSig.sol:351-358 — nested signature with `internalWeight >= internalThreshold` adds `externalWeight` — but if `internalWeight == internalThreshold` exactly, minor signature removal or threshold change can flip validation; off-by-one in threshold could allow 1-below-threshold signatures to pass via nested weight manipulation.
