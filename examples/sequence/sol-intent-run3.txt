## ACCESS CONTROL & INIT

- [ACCESS CONTROL] Hooks::addHook — onlySelf modifier — if wallet loses control or delegatecall path exists, attacker can add malicious hooks

- [ACCESS CONTROL] Stage1Auth/Stage2Auth — _isValidImage override — multiple inheritance paths could lead to unexpected validation behavior

## ORACLE / PRICE FEED

- None found after checking — no oracle or price feed integrations detected in this codebase

## ARITHMETIC

- [ARITHMETIC] LibBytes::readUintX — shift calculation `sub(256, mul(_length, 8))` — if _length > 32 underflow could corrupt data extraction
- [ARITHMETIC] Base64::encode — `shl(2, div(add(dataLength, 2), 3))` — potential overflow for very large dataLength inputs
- [ARITHMETIC] WebAuthn::verify — challenge/type index bounds — if clientDataJSON is malformed, `add(q, c)` could overflow and pass validation incorrectly
- [ARITHMETIC] SessionSig::recoverSignature — pointer arithmetic — multiple pointer increments without overflow checks could skip past buffer end
- [ARITHMETIC] PermissionValidator::validatePermission — cumulative value addition `value256 += previousUsage` — unchecked addition could overflow limit checks
- [ARITHMETIC] ExplicitSessionManager::_validateExplicitCall — `sessionUsageLimits.totalValueUsed += call.value` — unchecked addition could overflow valueLimit validation
- [ARITHMETIC] LibBytes::readUintX — `and(shr(shift, word), sub(shl(mul(8, _length), 1), 1))` — if _length causes shl overflow, mask becomes zero

## TOKEN ACCOUNTING

- [TOKEN ACCOUNTING] Calls::_execute — call.value sent without balance check — if wallet balance < call.value, execution continues with other calls potentially in inconsistent state

## Additional observations (HIGH RISK)

- **Nonce overflow at 2^256**: After `type(uint256).max` transactions in a space, nonce wraps to 0, enabling replay of original transactions
- **Static signature caller = address(0) bypass**: If `_setStaticSignature(hash, address(0), future)` is set, any caller passes the check `if (addr != address(0) && addr != msg.sender)`, allowing universal static sig
- **Chained signature downgrade**: Attacker can craft chain with valid current signature → transition to weak imageHash → weak imageHash signs malicious payload
- **Sapient signer reentrancy to poison weight**: Malicious sapient contract can call back into wallet during `recoverSapientSignature`, potentially reading stale storage
- **ecrecover address(0) = valid signer**: Invalid signature returns `address(0)` from ecrecover, but code treats it as valid address and adds weight
- **Hardcoded subdigest bypasses all security**: `FLAG_SUBDIGEST` and `FLAG_SIGNATURE_ANY_ADDRESS_SUBDIGEST` set weight to `type(uint256).max`, single leaf in tree overrides entire config
