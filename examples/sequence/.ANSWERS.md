## H-01 Chained signature with checkpoint usage disabled can bypass all checkpointer validation
When the wallet config's checkpointer is set but `useCheckpointer` is disabled on a chained signature, the checkpointer validation path is skipped entirely. A subsequent on-chain operation that relies on checkpointer-bound state can be authorized by a signature that should have failed the checkpointer check. Mechanism: `BaseSig`'s chained-signature path branches on the `useCheckpointer` flag before the checkpoint-enforcement call, never re-entering enforcement on the chained sub-signatures.

## H-02 Partial signature replay/frontrunning attack on session calls
A session signature for a multi-call batch can be replayed for a subset of the calls, or frontrun by extracting the inner per-call signatures. Mechanism: `SessionSig` validates each call's signature independently without binding to the batch identity / nonce / index, so an off-chain attacker can pick a profitable sub-call and submit it alone with the same signature material.

## M-01 Session signatures replay across wallets due to missing wallet binding
A session signature created for wallet A can be replayed against wallet B if both wallets registered the same session signer. Mechanism: `SessionManager` derives the signing payload from session config + call data without including `address(this)` (the wallet) in the EIP-712 domain or hashed payload, so the signature is wallet-independent.

## M-02 Static signatures bound to caller revert under ERC-4337, causing DoS
Static signatures keyed on `msg.sender` revert when the wallet is invoked via the ERC-4337 EntryPoint, since `msg.sender == EntryPoint` rather than the expected user-op submitter. Mechanism: `ERC4337v07` doesn't unwrap the caller identity for static-sig checks; wallets configured with caller-bound static signatures become unusable on 4337 paths.

## M-03 BaseAuth.recoverSapientSignature returns a constant instead of signer image hash, breaking sapient signer flows
`BaseAuth::recoverSapientSignature` returns a hard-coded constant rather than the recovered image hash of the sapient signer. Mechanism: function body ignores the recovered hash and returns a sentinel; all sapient-signer authorization paths short-circuit to the same value, conflating distinct signers.

## M-04 Factory deploy reverts instead of returning address when account already exists
`Factory::deploy` reverts when the deterministic address is already deployed instead of returning the existing address. Mechanism: the create2 path lacks an "is contract already deployed at this address?" pre-check; idempotent deploy callers (relayers, bundlers) crash on second invocation.

## L-01 Incorrect intermediate validation of cumulative parameter rules
Intermediate (per-call) cumulative-parameter checks in `PermissionValidator` use the wrong accumulator state, allowing values that would violate the rule when summed at the end to pass per-call validation.

## L-02 Value usage is incremented for fallback and aborted calls
Per-session value-spent counters in `ExplicitSessionManager` increment for fallback and reverted calls, prematurely exhausting session budgets.

## L-03 Nonce consumption reverts on execution failure enabling signature replay attacks
When inner execution reverts, the wallet's nonce is not consumed; the same signature can be replayed once the underlying revert condition is cleared.

## L-04 Unnecessary bitmasking in LibBytes::readUnitX
Defensive bitmasks in `LibBytes::readUnitX` are redundant given Solidity's calldata semantics; pure gas overhead.

## L-05 Inefficient usage limit increment call
Usage-limit accounting reads/writes the same slot multiple times per call where one read-modify-write would suffice.

## L-06 Duplicated delegate call validations
Delegate-call destination validation is performed in two locations in the call path; one can be removed.
