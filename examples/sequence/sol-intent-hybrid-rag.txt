# Findings

## ACCESS CONTROL & INIT

- ERC4337FactoryWrapper::deploy — Missing CREATE2 idempotency check before calling factory.deploy — If contract already exists at computed address, factory.deploy may revert but wrapper doesn't detect pre-existing contract consistently (checks code.length after, not before factory call), causing gas waste or re-entrancy risk if factory has side effects

- Stage1Module::constructor — Inherits from Stage1Auth which sets factory and nextModuleTarget in storage without access control — If initialization order changes or if base contracts are re-initialized, factory/nextModuleTarget could be overwritten by a different deployer

- Factory::deploy — No protection against front-running the deploy with same _mainModule and _salt — Attacker can observe mempool, deploy identical wallet with same salt, causing original deployer's transaction to revert or waste gas

- Estimator::_isValidImage — Override returns true for any imageHash without validation — Allows any signature to pass imageHash validation during gas estimation, potentially misleading estimators about valid configurations

- Recovery::queuePayload — No access control on who can queue a payload for a given wallet/signer pair — Griefing attack: anyone can queue arbitrary payloads for any wallet/signer, polluting storage and emitting misleading events

- Recovery::_recoverBranch — FLAG_RECOVERY_LEAF accepts any signer address from calldata — Malicious signature can include arbitrary signer addresses in recovery leaves, causing storage pollution if used in queuePayload

- SessionManager::recoverSapientSignature — No validation that payload.space is actually allocated to the caller — If space is not properly allocated, nonce consumption could collide with other modules or be bypassed

- SessionSig::recoverSignature — Identity signer is required but only validated to be non-zero, not that it matches a registered identity — If identity signer is changed after attestations are issued, old attestations could remain valid

- PermissionValidator::setLimitUsage — Internal function can be called by derived contracts without additional checks — If a malicious extension calls setLimitUsage, it could manipulate usage limits to bypass session restrictions

## ORACLE / PRICE FEED

- none found (no oracle or price feed integrations detected)

## ARITHMETIC

- ExplicitSessionManager::_validateExplicitCall — sessionUsageLimits.totalValueUsed += call.value without overflow check in unchecked context — If totalValueUsed + call.value overflows, user could bypass valueLimit by wrapping around to small value

- PermissionValidator::validatePermission — value256 += previousUsage without overflow check — If cumulative usage overflows, rule validation could incorrectly pass, allowing unlimited usage

- LibAttestation::fromPacked — pointer += dataSize without bounds check — If dataSize is maliciously large, pointer could overflow or read beyond calldata bounds

- LibBytes::readUintX — Shifts and masks arbitrary _length without validating _length <= 32 — If _length > 32, shift amount in sub(256, mul(_length, 8)) underflows, causing incorrect extraction

- SessionSig::recoverConfiguration — pointer increments in loop without total length validation — If encoded data is crafted to cause pointer to exceed calldata length, out-of-bounds reads could return dirty values

- Payload::fromPackedCalls — Decoding loop increments pointer without validating final pointer == _payload.length — Malformed payload could have trailing bytes that are silently ignored, causing signature mismatch

- WebAuthn::verify — assembly block computes sha256 with add(l, 0x20) without checking l + 0x20 overflow — If authenticatorData length is near max uint256, addition could wrap, causing incorrect hash

## TOKEN ACCOUNTING

- Guest::fallback — No validation that msg.value matches sum of call.value in _dispatchGuest — If msg.value < sum(call.value), calls will fail; if msg.value > sum, excess ETH is trapped in Guest contract

- Calls::_execute — call.value is sent without checking contract balance or msg.value — If wallet balance < call.value, call fails but no explicit check before iteration, wasting gas on invalid payloads

- ERC4337v07::validateUserOp — missingAccountFunds deposited to entrypoint without checking actual wallet balance — If wallet balance < missingAccountFunds, depositTo could fail or require external funding, breaking user op execution

- SessionManager::_validateExplicitCall — call.value > 0 increments totalValueUsed but no check that wallet has sufficient balance — User could sign a payload with call.value exceeding wallet balance, causing execution failure after signature validation

## ARRAY / INDEX / INPUT VALIDATION

- Simulator::simulate — _calls array length not validated before loop — If _calls.length is 0, function returns empty results array, potentially misleading callers

- SessionSig::recoverSignature — attestationCount read as uint8, limiting attestation list to 255 entries — If more than 255 attestations are needed, encoding/decoding will fail or truncate

- SessionSig::recoverConfiguration — permissionsCount used to resize sessionPermissions array without bounds check — If encoded data causes permissionsCount to exceed initial guess, array access could be out of bounds

- LibAttestation::fromPacked — dataSize read as uint24, limiting applicationData to 16MB — If applicationData exceeds 16MB, decoding will fail or truncate

- PermissionValidator::validatePermission — rule.offset used to read from call.data without validating offset < call.data.length — If offset is beyond calldata length, readBytes32 returns dirty values from scratch space

- Payload::fromPackedCalls — calls array allocated with numCalls from pointer without validating numCalls <= reasonable limit — If numCalls is maliciously large, allocation could cause OOM or excessive gas consumption

- Recovery::_recoverBranch — Recursive calls without depth limit — Deeply nested branches could cause stack overflow or excessive gas consumption

- ImplicitSessionManager::_isAddressBlacklisted — Binary search on unsorted blacklist could return incorrect results — If blacklist is not sorted, binary search may miss blacklisted addresses, allowing unauthorized calls

## EXTERNAL CALLS & REENTRANCY

- Calls::_execute — Delegate call to untrusted call.to without reentrancy guard on delegatecall — Malicious delegate could re-enter Calls::execute, bypassing nonce check or signature validation

- Guest::_dispatchGuest — LibOptim.call to untrusted call.to without CEI — Malicious callee could re-enter Guest::fallback, causing unexpected state changes

- Simulator::simulate — Delegate call to untrusted call.to in simulation context — Malicious delegate could modify simulator state or return misleading results

- Estimator::estimate — Delegate call to untrusted call.to in estimation context — Malicious delegate could consume excessive gas or return misleading gas estimates

- Recovery::isValidSignature — Low-level call to _signer.isValidSignature without return data size check — If _signer returns less than 32 bytes, comparison with IERC1271_MAGIC_VALUE_HASH could use dirty memory

- Passkeys::_decodeSignature — WebAuthn.verify performs staticcall to RIP_PRECOMPILE and VERIFIER — If precompile/verifier is not deployed or returns unexpected data, verification could incorrectly pass or fail

- SessionManager::recoverSapientSignature — ISignalsImplicitMode(call.to).acceptImplicitRequest is external view call — Malicious target could return false magic value, but more critically, could perform state-reading attacks to infer session details

- ERC4337v07::validateUserOp — IEntryPoint(entrypoint).depositTo called with missingAccountFunds — If entrypoint is malicious or paused, depositTo could fail or be front-run, causing user op validation to fail

## CROSS-CHAIN / BRIDGE

- SessionSig::recoverConfiguration — SessionPermissions.chainId checked against block.chainid but not enforced during signature recovery — If signature is replayed cross-chain, imageHash may differ but validation logic doesn't prevent replay if chainId is 0

- Payload::toEIP712 — noChainId flag allows omitting chainId from EIP712 domain — If noChainId is true, payload signed on one chain can be replayed on another chain with same wallet/nonce

- Recovery::recoveryPayloadHash — noChainId flag allows omitting chainId from EIP712 domain — Recovery payloads can be replayed cross-chain if noChainId is true

- SessionManager::recoverSapientSignature — payload.space checked against MAX_SPACE but not validated to match expected cross-chain nonce space — If nonce space is reused across chains, nonce replay could occur

## ECONOMIC / INCENTIVE LOGIC

- Recovery::queuePayload — No deadline or expiration on queued payloads — Old queued payloads remain valid indefinitely, allowing recovery after signers are rotated out

- SessionManager::_validateExplicitCall — sessionPermissions.deadline checked but no mechanism to revoke expired sessions from storage — Expired sessions remain in imageHash, requiring full configuration update to remove

- ExplicitSessionManager::incrementUsageLimit — Usage limits can only increment, never decrement — If a session is compromised, usage limits cannot be reset without changing the session configuration

- PermissionValidator::validatePermission — Cumulative usage limits persist across payloads but no mechanism to reset — If a session is long-lived, usage limits could reach max and block further calls, requiring session rotation

- SessionSig::recoverConfiguration — SessionPermissions.valueLimit is per-session, not per-call — If a session has multiple calls, total value sent could exceed intended limit if calls are in separate payloads

- ImplicitSessionManager::_validateImplicitCall — No expiration or deadline check on attestations — Attestations remain valid indefinitely, allowing implicit calls long after they should expire

- Recovery::_recoverBranch — requiredDeltaTime is validated but no maximum limit — Attacker could set requiredDeltaTime to max uint24 (194 days), delaying recovery indefinitely

- Passkeys::_rootForPasskey — metadata node included in root hash without semantic validation — Arbitrary metadata could be used to generate colliding image hashes if not uniquely scoped
