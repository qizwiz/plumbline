# Audit Findings

## Findings

### 1. ACCESS CONTROL & INIT

- [ACCESS CONTROL & INIT] IWithdrawRequestManager::setApprovedVault — missing access control specification — only owner should be able to approve vaults but interface lacks indication of protection
- [ACCESS CONTROL & INIT] IWithdrawRequestManager::rescueTokens — emergency exit role mentioned but not defined — if role is not properly initialized or checked, unauthorized rescue could occur

- [ACCESS CONTROL & INIT] IYieldStrategy::postLiquidation — no access control specified — unauthorized post-liquidation could corrupt accounting after liquidation completes
- [ACCESS CONTROL & INIT] IYieldStrategy::clearCurrentAccount — no access control specified — anyone clearing current account could break CEI patterns or reentrancy guards

### 2. ORACLE / PRICE FEED

- [ORACLE / PRICE FEED] IYieldStrategy::convertYieldTokenToAsset — no staleness check mentioned — stale oracle price could allow liquidations at unfair prices or borrowing beyond safe limits
- [ORACLE / PRICE FEED] IYieldStrategy::price — takes borrower into account but no sequencer uptime check — on L2s, sequencer downtime could cause stale prices during liquidations
- [ORACLE / PRICE FEED] IOracle::price — returns uint256 with no bounds checking — oracle returning very high price could overflow in maxBorrow or assetsRepaid calculations as warned in IMorpho comments
- [ORACLE / PRICE FEED] IYieldStrategy::convertYieldTokenToAsset — no decimals normalization mentioned — if yieldToken and asset have different decimals, conversion could be orders of magnitude wrong

### 3. ARITHMETIC

- [ARITHMETIC] WithdrawRequest::yieldTokenAmount — uint120 storage — if yield token amounts exceed 2^120, overflow will silently wrap causing catastrophic accounting errors
- [ARITHMETIC] WithdrawRequest::sharesAmount — uint120 storage — if shares amount exceeds 2^120 during high-leverage scenarios, overflow causes loss of user claims
- [ARITHMETIC] TokenizedWithdrawRequest::totalYieldTokenAmount — uint120 storage — accumulated yield token amounts could overflow uint120 over time with many users
- [ARITHMETIC] TokenizedWithdrawRequest::totalWithdraw — uint120 storage — total withdraw could exceed uint120 in high-TVL vaults causing wrap-around
- [ARITHMETIC] IYieldStrategy::convertToShares — no rounding direction specified — if rounds down when minting shares, protocol loses assets; if rounds up when burning, users lose assets
- [ARITHMETIC] IYieldStrategy::convertToAssets — no rounding direction specified — asymmetric rounding could systematically favor users over protocol or vice versa
- [ARITHMETIC] IWithdrawRequestManager::finalizeAndRedeemWithdrawRequest — sharesToBurn could exceed sharesAmount — unchecked subtraction could underflow if accounting is inconsistent

- [TOKEN ACCOUNTING] IWithdrawRequestManager::finalizeAndRedeemWithdrawRequest — tokensWithdrawn returned but no check for actual transfer — if withdraw token transfer fails silently, accounting diverges from balances

- [TOKEN ACCOUNTING] IYieldStrategy::burnShares — assetsWithdrawn returned but no verification — if withdrawal partially fails, user credited more assets than actually sent
- [TOKEN ACCOUNTING] IYieldStrategy::asset — no check if asset is standard ERC20 — non-standard tokens omitting return values could cause silent transfer failures
- [TOKEN ACCOUNTING] IYieldStrategy::totalAssets — includes compounding from yield but no cap — if totalAssets grows above 1e35 as warned, toSharesUp/Down calculations overflow

### 6. EXTERNAL CALLS & REENTRANCY
