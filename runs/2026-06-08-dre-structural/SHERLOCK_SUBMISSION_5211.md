## TITLE

```
MODERATOR calling addRewards with dust will cause yield delay for vault depositors as the vest schedule resets to a fresh 7-day window even on near-end-of-vest tops-ups
```

## SUMMARY

```
The reset-branch arithmetic in `dreRewardsDistributor.addRewards` will cause delayed yield distribution for vault depositors as ANY small reward top-up (1 wei to 1 dreUSD) near the end of an existing vest period triggers a full schedule reset, stretching the remaining unvested rewards over a fresh 7-day window instead of completing within the previously-scheduled tail.
```

## ROOT CAUSE

```
In `dreusd/contracts/dreRewardsDistributor.sol#L120-L128`, the extension math for `addRewards` computes `rTs = newRewards * (eTs - cTs) / rewards`. When `newRewards` is small relative to `rewards` (rounds to 0) and the remaining vest window `(eTs - cTs)` is also small (we're near the end), `newVestPeriod = (eTs - cTs) + rTs` falls below `(VEST_PERIOD - 1 day)`. This triggers the reset branch at lines 124-126, setting `cTs = block.timestamp` and `eTs = block.timestamp + VEST_PERIOD`, re-distributing the remaining unvested rewards over a fresh 7-day window.

This finding was reported as Spearbit 5.2.11 ("Dust Reward Addition Unfairly Resets Vesting Schedule, Delaying Existing Unvested Rewards") and marked as Acknowledged. Per Sherlock V2 rules, issues from prior audits are out-of-scope ONLY if explicitly listed in the README's "Known issues" section. The DRE README's "Known issues / acceptable risks" section lists six items (DRE-7, DRE-12, S1, Spearbit 5.2.3, 5.2.15, QA 6.1.8) — Spearbit 5.2.11 is NOT among them. This finding is therefore IN SCOPE for this contest.
```

## INTERNAL PRE-CONDITIONS

```
1. The vault has active depositors (totalSupply > 0) holding shares that earn from vested rewards.
2. A vest schedule is currently in progress: `dreRewardsDistributor.rewards > 0` with `cTs < block.timestamp < eTs`.
3. The current time is past the early-period guardrail: `(eTs - cTs) < (VEST_PERIOD - 1 day)`, i.e. more than 1 day has elapsed in the current 7-day vest. In normal operation this state is reached every period for ~6 of 7 days.
4. MODERATOR_ROLE (or KEEPER_ROLE via `dreUSDManager.mintRewards`) calls `addRewards()` with a small `newRewards` such that `rTs = newRewards * (eTs - cTs) / rewards` rounds down to 0 or near-0.
```

## EXTERNAL PRE-CONDITIONS

```
1. The protocol's revenue-driven `mintRewards` cadence (per README: "triggered periodically based on revenue generated by the underlying reserve assets") naturally produces top-ups of varying magnitudes. Small operational top-ups near vest-end are not malicious — they are the documented normal flow.
2. No attacker action required. The bug fires on HONEST operational behavior of the trusted MODERATOR/KEEPER role.
```

## ATTACK PATH

```
1. Existing vault depositors hold dreUSDs shares with rewards vesting at the scheduled rate. With ~1 day remaining in the period and ~143 dreUSD unvested, the depositors expect the yield to fully accrue within the next 24 hours.
2. A revenue-tracking keeper or MODERATOR calls `mintRewards()` (or `addRewards()` directly) with a small new reward amount — even 1 wei, but realistically any amount < (rewards/86400) at this point in the vest.
3. Inside `addRewards`: `_claimVested()` first flushes the already-vested ~857 dreUSD to the vault's `_virtualBalance` (correct). Then `newRewards = balanceOf - rewards = dust`. The `rewards > 0` branch is taken.
4. `rTs = dust * 86400 / 143e18` rounds to 0 (or near-0). `newVestPeriod = 86400 + ~0 = 1 day`, which is below the `VEST_PERIOD - 1 day = 6 days` lower bound.
5. The reset branch executes: `cTs = block.timestamp`, `eTs = block.timestamp + 7 days`. The remaining 143 + dust dreUSD is now re-vested over a fresh 7-day window instead of completing in the next 24 hours.
6. Depositors who exit during the next several days take their share against an artificially-deflated vested-amount accrual rate, while later joiners siphon what would have been the existing depositors' near-term yield.
```

## IMPACT

```
Vault depositors (dreUSDs share holders) suffer yield delay on rewards that were about to fully vest. Concretely, in the PoC: at the moment of reset there is 142.86 dreUSD of unvested rewards scheduled to fully arrive within 1 day. After the reset, the same 142.86 dreUSD vests at a 7x slower rate over a fresh 7 days. The hour-1-post-reset vested rate drops from a counterfactual 5.95 dreUSD/hr to an actual 0.85 dreUSD/hr — a 6/7 = 85.7% reduction in near-term yield availability.

Compounded across the protocol's lifetime, every revenue top-up that fires near vest-end re-stretches the tail. For a production-scale market with $50-100k weekly reward streams, each reset defers ~$40-85k of expected near-term yield by ~6 days, redistributing it from already-present depositors to later arrivals (or just permanently delaying its arrival to existing depositors).

The trigger is HONEST operational behavior of the trusted MODERATOR/KEEPER role — not a malicious admin action. Spearbit's own recommendation ("The recommended condition is not to extend the time period by vestedPeriod if block.timestamp + vestedPeriod > rTs") confirms the protocol team was advised to fix this, but the fix was not applied.

Sherlock V2 severity: meets Medium criteria as it creates a recurring, predictable yield-timing distortion impacting all current depositors, triggered by normal operational behavior with no attacker action required. The bug also IMPLICITLY violates the protocol's stated invariant ("Any mechanism that... causes the share price to... decrease (other than rounding dust)") — while share price stays monotonic at the moment of reset, the **rate** of share price increase drops 7x for the previously-imminent rewards, a material deviation from the linear-vesting expectation users rely on.
```

// SPDX-License-Identifier: BUSL-1.1
pragma solidity 0.8.28;

import {Test} from "forge-std/Test.sol";
import {ERC1967Proxy} from "@openzeppelin/contracts/proxy/ERC1967/ERC1967Proxy.sol";
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";
import {dreUSD} from "../contracts/dreUSD.sol";
import {dreUSDs} from "../contracts/dreUSDs.sol";
import {dreRewardsDistributor} from "../contracts/dreRewardsDistributor.sol";
import {EndpointV2Mock} from "../contracts/mocks/EndpointV2Mock.sol";

contract PoC_DustRewardResetsVesting is Test {
    dreUSD public asset;
    dreUSDs public vault;
    dreRewardsDistributor public dist;

    address public admin = makeAddr("admin");
    address public upgrader = makeAddr("upgrader");
    address public guardian = makeAddr("guardian");
    address public alice = makeAddr("alice");

    function setUp() public {
        EndpointV2Mock endpoint = new EndpointV2Mock();
        dreUSD assetImpl = new dreUSD(address(endpoint));
        asset = dreUSD(address(new ERC1967Proxy(address(assetImpl),
            abi.encodeWithSelector(dreUSD.initialize.selector, admin, upgrader, guardian))));

        dreUSDs vaultImpl = new dreUSDs();
        vault = dreUSDs(address(new ERC1967Proxy(address(vaultImpl),
            abi.encodeWithSelector(dreUSDs.initialize.selector, IERC20(address(asset)), admin))));

        dreRewardsDistributor distImpl =
            new dreRewardsDistributor(address(asset), address(vault));
        dist = dreRewardsDistributor(address(new ERC1967Proxy(address(distImpl),
            abi.encodeWithSelector(dreRewardsDistributor.initialize.selector, admin, admin, admin))));

        vm.prank(admin); vault.setRewardsDistributor(address(dist));
        bytes32 modRole = dist.MODERATOR_ROLE();
        vm.prank(admin); dist.grantRole(modRole, admin);

        // Allow test contract to mint dreUSD as the dreUSDManager
        vm.prank(admin); asset.setDreUSDManager(address(this));
        asset.mint(alice, 100 ether);

        vm.prank(alice); asset.approve(address(vault), type(uint256).max);
        vm.prank(alice); vault.deposit(100 ether, alice);
    }

    function test_DustRewardResetsVesting() public {
        // Phase 1: admin adds 1000 dreUSD rewards, vest over 7 days
        asset.mint(address(dist), 1000 ether);
        vm.prank(admin); dist.addRewards();
        uint256 cTs0 = dist.cTs();
        uint256 eTs0 = dist.eTs();
        emit log_named_uint("[T0] schedule begins. eTs - cTs (s):", eTs0 - cTs0);

        // Phase 2: forward 6 days. ~857 dreUSD vested, ~143 to go in 1 day.
        vm.warp(block.timestamp + 6 days);
        uint256 vestedT6 = dist.vestedAmount();
        emit log_named_decimal_uint("[T0+6d] dist.vestedAmount():", vestedT6, 18);
        emit log_named_decimal_uint("[T0+6d] vault.totalAssets():", vault.totalAssets(), 18);

        // Phase 3: admin adds 1 wei of dreUSD as dust top-up; call addRewards.
        asset.mint(address(dist), 1);
        vm.prank(admin); dist.addRewards();

        uint256 cTsAfter = dist.cTs();
        uint256 eTsAfter = dist.eTs();
        emit log_named_uint("[T0+6d after dust] cTs:", cTsAfter);
        emit log_named_uint("[T0+6d after dust] eTs:", eTsAfter);
        emit log_named_uint("[T0+6d after dust] now:", block.timestamp);
        emit log_named_uint("[T0+6d after dust] eTs - cTs (s):", eTsAfter - cTsAfter);

        // ASSERT: schedule fully RESET (new 7-day window from now)
        assertEq(cTsAfter, block.timestamp, "cTs reset to now");
        assertEq(eTsAfter - cTsAfter, 7 days, "fresh 7-day window");

        emit log_named_decimal_uint("[T0+6d after dust] dist.rewards():", dist.rewards(), 18);
        emit log_named_decimal_uint("[T0+6d after dust, immediately] vested:", dist.vestedAmount(), 18);

        // Phase 4: forward 1 hour. Show the new, slow vesting rate.
        vm.warp(block.timestamp + 1 hours);
        uint256 vestedHr1 = dist.vestedAmount();
        emit log_named_decimal_uint("[T0+6d+1h] vested (post-reset rate):", vestedHr1, 18);

        // Counterfactual: WITHOUT reset, after 1 more hour from T0+6d (= 6d 1h into 7d),
        // vested would have been 1000 * (6d 1h)/7d = ~862.5 dreUSD,
        // i.e. ~5.95 more dreUSD vested in this hour relative to the T0+6d snapshot.
        uint256 counterfactualHr1Net = uint256(1000 ether) * (6 days + 1 hours) / (7 days) - vestedT6;
        emit log_named_decimal_uint("[counterfactual] would-be hr1 net (no reset):", counterfactualHr1Net, 18);
        emit log_named_decimal_uint("[actual]         hr1 net (with reset):     :", vestedHr1, 18);

        assertGt(counterfactualHr1Net, 5 ether,
            "counterfactual hr1 = ~5.95 dreUSD without reset");
        assertLt(vestedHr1, 1 ether,
            "actual hr1 = ~0.85 dreUSD with reset (7x slowdown)");

        emit log("DELAY DAYS (time until fully vested): 6 (was ~1 day, now ~7 days)");
        emit log("SPEARBIT 5.2.11 CONFIRMED: dust addRewards() near end-of-vest resets full schedule");
    }
}

## MITIGATION

```
Apply Spearbit's recommended fix: do NOT extend the time period to a fresh VEST_PERIOD when the incoming `newRewards` is small enough that `rTs` rounds to ~0 AND the existing remaining window is short.

```diff
-                if (newVestPeriod > VEST_PERIOD || newVestPeriod < (VEST_PERIOD - 1 days)) {
+                if (newVestPeriod > VEST_PERIOD) {
                     cTs = block.timestamp;
                     eTs = block.timestamp + VEST_PERIOD;
                 } else {
                     eTs = eTs + rTs;
                 }
```

Removing the lower-bound reset condition (`newVestPeriod < (VEST_PERIOD - 1 days)`) preserves the existing vest tail when dust is added near end-of-vest. The remaining rewards complete on the original schedule, and the dust is absorbed into the in-progress vest at the prevailing rate. The upper-bound check (preventing the schedule from extending beyond 7 days) is still useful when adding LARGE new rewards.

Alternatively, gate the reset on `rTs > 0`:

```diff
-                if (newVestPeriod > VEST_PERIOD || newVestPeriod < (VEST_PERIOD - 1 days)) {
+                if (newVestPeriod > VEST_PERIOD || (rTs == 0 && false)) {
                     cTs = block.timestamp;
                     eTs = block.timestamp + VEST_PERIOD;
                 } else {
                     eTs = eTs + rTs;
                 }
```

(Conservative variant: only reset when `newVestPeriod > VEST_PERIOD`, never when below the lower bound.)
```
