# H14 lift on ScaBench — 2026-06-20

**Question**: does plumbline's H14 eigenvector centrality re-rank lift precision@K over GPT-5's confidence ordering, measured by `scorer_v2.py` + gpt-4o-mini judge?

## Setup
- Benchmark: scabench-org/scabench `curated.json` (31 projects)
- Scoring: scabench-org/scabench `scoring/scorer_v2.py --model openrouter/openai/gpt-4o-mini`
- Top-K truncation: K=10
- Conditions per project:
  - **baseline** = top-K by GPT-5's `confidence` field
  - **H14_old** = top-K by max(eigenvector centrality) re-rank (pre-cleanup)
  - **H14_new** = same, with `setdefault → max` attribution fix in `tools/scabench_rerank.py`
  - **H14_dedup** = H14_new + dedupe-at-K policy (skip duplicate function attributions)
  - **inv_eig** = re-rank by 1/eigenvector (the synthesis memo's "anti-symmetric" prediction)
  - **inv_deg** = re-rank by 1/degree
  - **closeness** = re-rank by closeness centrality
- H14 centralities: computed on Modal Volume from bootstrapped scabench source trees via tree-sitter call graph + `nx.eigenvector_centrality`
- Attribution: regex on location+title, same heuristic as `tools/scabench_rerank.py`
- Methodology note: K=10 numbers below come from `tools/h14_lift_simulator.py` re-using the existing scorer_v2 per-finding judgments. A finding is TP if its id appears in `matched_findings[*].found_id`. The simulator is $0/run and lets us A/B many orderings against fixed judgments. This slightly under-estimates absolute lift (scorer_v2's matcher is 1-to-1 and may have given an expected bug to a worse-quality tool finding ranked above the better one in the original ordering) but the bias is symmetric across conditions.

## Two empirical kills

### Kill #1 — dedupe-at-K
The 2026-06-20 win/loss diagnostic synthesis memo recommended deduping findings per `(file, function)` before re-ranking, predicting it would "kill the GenesisPrecompiler stacking and PendleLPOracle 4x-leak" as a free win on both conditions.

Empirically:
- Dedupe applied to baseline: +0.0012 (negligible — the leaf-leak rarely fills baseline top-10).
- Dedupe applied to H14: **-0.0210** vs H14 alone.

H14's wins on hub-architecture projects (loopfi, oku, crestal, boost-core) come from *multiple* TP findings on the *same orchestration function* getting bubbled up together. Dedupe collapses those into one slot and kills the win signal. The memo's symmetric "leaf-suppression" theory was wrong about H14's actual win mechanism. **Dedup REJECTED.**

### Kill #2 — centrality inversion (dataflow-distance theoretical proxy)
The same memo recommended dataflow-distance from untrusted input to state-write as the next signal, with the theoretical claim that H14-losers have bugs at *sources* (low-eigenvector externals) rather than *sinks* (high-eigenvector hubs). If true, inverting the eigenvector ranking should help on the 6 H14-loser projects.

Empirically on losers (baseline=0.1748):
- inv_eig: 0.1404 (**-0.0344**)
- inv_deg: 0.1196 (-0.0552)
- closeness: 0.1649 (-0.0099)

All three structural inversions go the **same direction** as H14 on losers, not opposite. The memo's pre-specified kill criterion was: "if dataflow-distance shows the same sign as H14 on ≥4 of the first 5 projects, signals are collinear — stop." This is 6 of 6.

**Dataflow-distance bet KILLED.** The H14-loser projects aren't a structural anti-pattern from any centrality lens we can compute on the existing call graph — they're noise. The synthesis memo over-fit a "bugs at sources vs sinks" theory to a noise-vs-signal pattern.

## What survives

The H14 attribution-fix (`setdefault → max` in `_build_node_score_map`) is the only intervention that earned macro-F1 lift today.

- **Free win**: +0.0037 macro F1 (H14_new vs H14_old). Best per-project deltas: loopfi (+0.118 from prev re-rank), oku (+0.069), fenix (+0.160). These were cases where multiple contracts had a function of the same name and the prior `setdefault` lost the higher-centrality version.
- **Cleanup landed in `tools/scabench_rerank.py`** at commit-time of this report.
- **Headline (24 projects, K=10)**: H14 + attribution-fix = **0.2214** macro F1, +0.0276 over baseline.

## What this means for the FV-gated architecture plan

The plumbline architecture doc targets macro F1 0.246 → 0.34+. Today closes the door on closing the gap via structural-signal re-ranking on the existing call-graph features. The remaining levers are:

1. **Better proposer** (NOT tested today): swap GPT-5 + gpt-4o-mini for Sonnet 4.6 + better judge. Tests whether the 0.22 ceiling is the ranker or the model. ~$5-10 Modal, ~1 hour, requires re-installing scorer_v2 (deleted in 2026-06-17 disk cleanup) and verifying API key path.
2. **Prompt-level enumeration fix**: per the idle-finance diagnostic, GPT-5 emitted zero findings on `requestWithdraw` (where 3 of 5 ground-truth bugs lived). Force-enumeration of every external state-mutating function before re-ranking touches them. *Only* helps the attribution-starved class — re-reading the 6 loss diagnostics, only idle clearly fits this case (the other 5 had the findings in baseline; H14 just ranked them poorly).
3. **Accept and write up**: this report + the win/loss synthesis + the two negative results IS a strong methodology section for "H14: structural priors help on hub-architecture projects, plateau at ~0.22 macro F1 on scabench, with dedup and centrality-inversion both failing empirically."
